# Spec-Driven Workflow

A generic, cross-tool (Claude Code + Cursor) agent workflow for spec-driven development. Replaces chat-held intent with repo-held specs. Every feature starts with a spec, every implementation is verified, every handoff is auditable.

## Commands

| Command | Purpose |
|---------|---------|
| `/plan` | Create or iterate spec + phased implementation plan |
| `/implement` | Execute plan phase by phase with spec echo and verification |
| `/verify` | Run acceptance checks and cross-check completeness |
| `/investigate` | Evidence-first diagnosis without speculative patches |
| `/commit` | Atomic git commits with conventional messages |
| `/pr` | Generate PR description enriched from specs |
| `/research` | Deep parallel codebase exploration |
| `/invariants` | Analyze codebase for correctness rules |
| `/tests` | Generate high-quality tests from invariants |
| `/handoff` | Write structured session handoff |
| `/resume` | Restore context from spec artifacts |
| `/bootstrap` | Scaffold new repo to agent-ready state |

## Spec Structure

Each feature/task gets a folder under `specs/<slug>/`:

- `spec.md` — IntentSpec YAML: objective, outcomes, constraints, edge cases
- `tasks.md` — Phased plan with success criteria
- `acceptance.md` — Exit criteria and verification commands
- `research.md` — Optional investigation findings
- `verification.md` — Machine-written proof (auto-generated by /verify)
- `handoff.md` — Session transition document

Templates for these files live in `templates/` within this skill.

## Source of Truth

1. `CLAUDE.md` / `AGENTS.md` — repo-wide rules
2. `specs/<slug>/spec.md` — feature intent
3. `specs/<slug>/acceptance.md` — exit criteria
4. Verification artifacts — machine proof
5. `specs/<slug>/research.md` — optional context
6. Chat history — ephemeral, never authoritative

## Usage

Start with `/plan` for any non-trivial work. The command adapts:
- No spec exists → creates spec + plan (scoping mode)
- Spec exists without plan → researches and plans
- Plan exists → enters iteration mode

Then `/implement` → `/verify` → `/commit` → `/pr`.

For bugs, use `/investigate` first (evidence before patches).

For new repos, use `/bootstrap` to scaffold the structure.
